Summary

It is reading notes about the book – SCJP.

Assignments

  1. Stack v.s Heap

    Stack: Local variable(method variable/automatic) lives

    Heap: Objects and instance variable live

  2. Primitive Casting

    a. narrowing (put a larger thing (like long) into a smaller containers (like short)) requires explicit cast

    b. eg:

    implicit cast:
int a = 100;
long b = a; // an int value always fits in a long
eplicit cast:
float a = 100.001f;
int b = (int)a; //the float could lose info

  3. Arrays

    a. array elements are always given default values regardless of where the array itself declared or instantiated

    b. JVM only needs to know the size of array (the following eg is legal)

    int[][] array = new int[3][];

    c. primitive array can accept any value that can be promoted implicitly to the declared type of the array.

    eg:
int[] wl = new int[3];
byte b = 4;
char c = 'c';
short s = 7;
wl[0] = b; //ok, byte is smaller than int
wl[1] = c; //ok, char is smaller
wl[2] = s; //ok, short is smaller

    d. Arrays of object reference

     i. array element could be subclass of declared type
 ii. if array is declared as an interface, array element could refer to any instance of class that implements the declared interface

  4. Java is passing by value (a copy of the bits representing the reference)

  5. Wrapper class

    a. corelates to primitives

    b. two fuctionalities

      i. primitives can be handled like objects
  ii. provide unitility methods for primitives(usually conversions)

    c. Wrapper objects are immutable

    d. All wrapper class(except Character) provides 2 constructors:

      i. primitive of the type being constructed
  ii. String representation of the type being constructed
  eg:
  Integer i1 = new Integer(2);
  Integer i2 = new Integer("2");


Exception: Character, only provides 1 constructor, which takes char argument

  6. Overloading tricks

    when an exact method isn’t found, what method JVM looks for?

     a.widening (method with smallest arguement that is wider than the parameter)
 b. autoboxing (int, autoboxing to Integer)
 c. var-args (int, int ...x)

    eg:

     widening v.s autoboxing
 class AddBoxing {
 static void go(Integer x) {System.out.println("Integer");}
     static void go(long x) {System.out.println("long");}
     public static void main(String[] args) {
         int i = 5;
         go(i); //output: long---> widening wins
}
   }    
 autoboxing v.s var-args
  class AddVarargs {
      static void go(Byte x) {
        System.out.println("Byte");
     }
     static void go(byte ...x) {
        System.out.println("byte");
 }
     public static void main(String[] args) {
        byte b = 3;
        go(b);// output: Byte --> autoboxing wins
 }
   }

    d. none of wrapper classes will widen from one to another

  7. Garbage Collection(GC)

    a. Java manages memory automatically

    b. heap is part of memory where Java obj lives, and it’s the only part involved in garbage collection process.

    c. JVM decides when GC run (typically when sense memory running low)

    d. how GC work: mark and sweep algorithm

    e. when an object is eligible for GC: when no live thread can access it

    f. GC cannot guarantee enough memory: if maintain too many live objects –> out of memory

    g. writing code that explicitly make object eligible for collection:

     i. nulling a reference
 ii. reassign a reference variable
 iii. isolate a reference (two instance of same class reference to each other, so no alive thread can access)
 iv. force GC (System.gc()) // cannot guarantee 
    (reason: 1). JVM may not have implemented this routine (the languagespecification allows this routine to do nothing at all; 2). another thread might grab lots of memory right after you run GC))

    fyi.

    The GC routines that Java provides are members of the Runtime class(special class that has a single object(singleton) for each program);

    The Runtime object provides a mechamism for communicating directly with Virtual Machine, Runtime.getRuntime() –> returns Runtime instance (the Singleton)

    h. finalize():

     i. allows run some code before object deleted (clean up resources)
 ii. only called once(at most!) by GC
 iii. not guaranteed to run, don't put important code
 iv. may result in save an object from deletion
